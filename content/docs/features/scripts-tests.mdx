Every request can run JavaScript before sending (pre-script) and after receiving (post-script / tests). Executed by **JavaScriptCore** — the same engine Safari uses. Sandboxed, no Node access.

## API surface

Koala exposes a `koala` global. Postman scripts often work unchanged with `pm` → `koala` rename.

| Object | Purpose |
|---|---|
| `koala.request` | Read/mutate the outbound request (URL, method, headers, body) |
| `koala.response` | Read-only — status, headers, body, time |
| `koala.env` | Read/write environment variables (persisted) |
| `koala.globals` | Read/write global vars (across envs) |
| `koala.vault` | **Read-only** — keychain secrets, never returns to script logs |
| `koala.test()` | Register an assertion |
| `koala.expect()` | Chai-style fluent assertions |

## Pre-request script

Runs before the request is sent. Use it to compute a signature, refresh a token, generate a nonce.

```js
// Pre-script: HMAC-sign the request body
const ts = Date.now();
const body = koala.request.body || '';
const secret = koala.vault.get('signing_secret');
const sig = koala.crypto.hmacSha256(ts + body, secret);

koala.request.headers.set('X-Timestamp', ts);
koala.request.headers.set('X-Signature', sig);
```

Built-in libs available:

- `koala.crypto` — hmacSha256, sha256, md5, base64.
- `koala.uuid()` — v4 UUID.
- `koala.now()` — Date.now() shortcut.
- Standard `Math`, `JSON`, `Date`, `RegExp`, `String`, `Array`, `Object`, `Promise`.

## Tests / post-script

Runs after the response. Make assertions, persist data for the next request.

```js
koala.test('status is 201', () => {
  koala.expect(koala.response.status).to.equal(201);
});

koala.test('body has user id', () => {
  const body = koala.response.json();
  koala.expect(body.id).to.be.a('string');
  koala.expect(body.email).to.match(/@/);
});

// Capture for the next request
const token = koala.response.json().accessToken;
koala.env.set('auth_token', token);
```

Results appear in the **Tests** tab of the response panel. Pass/fail count badge on the tab.

## Chained scripts in collections

When running a collection (or via [scheduler](/docs/features/automation)):

- Each request's pre-script runs first.
- After response, post-script runs.
- A request can set env/global vars that the next request reads (e.g. `auth_token` flow).
- Test failures don't stop the run by default — toggle "Stop on failure" in run config.

## Collection runner

Sidebar → right-click collection → **Run…**. Or `⌘⇧R`.

```
✓ POST /auth/login         201  88ms   (3 tests passed)
✓ GET  /users/me           200  31ms   (2 tests passed)
✗ POST /users              400  20ms   (1 of 3 tests failed)
✓ DELETE /users/123        204  18ms   (1 test passed)
```

Options:

- **Iterations** — repeat N times. Use with data-driven runs (CSV / JSON input file).
- **Delay** — ms between requests (rate-limit friendly).
- **SLA budget** — fail run if total time exceeds X ms.
- **Stop on failure** — bail on first failure.
- **Webhook on failure** — POST run summary to a URL (works with Slack incoming webhook).
